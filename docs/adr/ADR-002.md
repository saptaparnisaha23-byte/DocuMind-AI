# ADR-002: Selection of Google Gemini API as Primary LLM Provider & Resilient Fallback Cascade

## Status
Accepted

## Context
DocuMind-AI requires a high-performance Large Language Model (LLM) to synthesize context-grounded answers from document passages retrieved during the RAG process. Key requirements for the LLM include:
1. High reasoning capability and precise instruction-following (for strict citation enforcement).
2. Low generation latency for real-time user interaction.
3. Cost efficiency and high API quota limits suitable for cloud deployment.
4. Resilient handling of API rate limits (HTTP 429 quota exhaustion).

## Decision
We chose **Google Gemini API** (`gemini-2.5-flash` / `gemini-1.5-flash`) as the primary LLM provider, combined with an automated multi-tier fallback cascade mechanism.

When an API call is initiated:
1. The system sends the prompt and context passages to `gemini-2.5-flash`.
2. If HTTP 429 (rate limit exceeded) occurs, the engine automatically cascades to `gemini-1.5-flash`.
3. If all remote endpoints fail or the environment is offline, the pipeline falls back to local context extraction.

## Consequences
### Positive
- **High Accuracy & Grounding**: Gemini models follow system prompts strictly, refusing to answer out-of-corpus queries and providing accurate inline page citations.
- **Low Latency & High Throughput**: Fast token generation times (<1.5s response latency).
- **High Reliability**: The fallback cascade eliminates single-point-of-failure risks caused by API quota limits.

### Negative
- **External Network Dependency**: Requires active internet connectivity and valid API key credentials for full generation capabilities.

## Alternatives Considered
- **OpenAI GPT-4o-mini**: Rejected due to lower free-tier rate limits and higher cost per 1M tokens.
- **Local Llama 3 / Qwen 2.5 via Ollama**: Rejected due to high RAM and GPU memory requirements when deployed on free-tier cloud platforms.
