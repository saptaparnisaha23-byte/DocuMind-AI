# ADR-003: Reciprocal Rank Fusion (RRF k=60) & Hybrid BM25 Vector Retrieval Engine

## Status
Accepted

## Context
Standard Retrieval-Augmented Generation (RAG) applications rely solely on dense vector search (e.g., cosine similarity over Sentence Transformer embeddings). However, dense retrieval frequently misses exact keyword terms, acronyms (e.g., MLR, SLR, OLS), table names, or specific numerical codes present in technical documents. Conversely, sparse keyword search (BM25) fails on paraphrased queries or conceptual synonyms.

## Decision
We implemented a **Hybrid Retrieval Engine** combining dense vector search (ChromaDB + `all-MiniLM-L6-v2`) and sparse keyword matching (Rank-BM25), fused via **Reciprocal Rank Fusion (RRF)** with constant $k=60$.

The RRF score for each document chunk $d$ is calculated as:
$$RRF\_Score(d) = \sum_{m \in M} \frac{1}{k + r_m(d)}$$
where $M$ is the set of retrieval models (Dense Vector and BM25 Sparse), and $r_m(d)$ is the rank position of chunk $d$ in model $m$.

## Consequences
### Positive
- **Scale-Invariant Merging**: RRF merges rank positions rather than raw distance scores, eliminating score calibration issues between Cosine Distance (-1 to 1) and BM25 (0 to +inf).
- **Superior Retrieval Recall**: Captures both deep semantic concepts and exact keyword/acronym matches.
- **Robustness**: Acronym expansion logic further boosts exact keyword match recall for domain-specific technical documents.

### Negative
- Slight increase in memory usage to maintain the in-memory BM25 index alongside ChromaDB.

## Alternatives Considered
- **Pure Dense Vector Search**: Rejected due to frequent misses on specific technical acronyms and verbatim search terms.
- **Weighted Linear Score Combination**: Rejected because Cosine similarity and BM25 scores have incompatible scale distributions that require constant manual tuning.
